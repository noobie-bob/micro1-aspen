The seeded vulnerabilities in this project-management API span 12 OWASP classes,
all centered on a Flask service (~1.5k LOC) managing teams, projects, tasks,
comments, and attachments across 9 route modules and 25+ endpoints.

Vulnerability classes:
1. Cross-team data isolation failure (BOLA/IDOR)
2. Missing field redaction on admin-only metadata
3. SQL injection via raw string interpolation
4. OS command injection via shell=True
5. Mass assignment on PATCH endpoints
6. Sensitive data exposure via unauthenticated debug endpoints
7. Information exposure via verbose error handlers
8. SSRF via unvalidated webhook URLs
9. Weak cryptography on share-link tokens
10. Path traversal on attachment downloads
11. Improper admin endpoint gating
12. Deprecated/legacy API endpoints without authentication

Test-authoring is the right shape because the threat surface is broad (25+
endpoints, 9 modules) and requires multi-step reasoning to discover — the
source code contains NO BUG annotations or vulnerability labels.

Rubric design (v5):
- 3 critical items (×5): cross-team list isolation, SQL injection, debug endpoint lockdown
- 22 major items (×3): remaining direct-coverage + regression guards
- 2 minor items (×2): path traversal, secondary coverage
- 1 nitpick item (×1): test quality / sentinel markers

Severity distribution: 3×5 + 22×3 + 2×2 + 1×1 = 15 + 66 + 4 + 1 = 86
rubric_max_score = 86

Rubric descriptions are behaviourally abstract — they describe the observable
behaviour the test should verify without naming specific fields, endpoints, or
vulnerability mechanisms. This is critical because the agent reads task_config.json.

Production image contains ONLY the substrate and conftest.py. The agent writes
ALL test files from scratch, including smoke/regression tests.
