A teammate on the platform noticed concerning behaviour in the ProjHub API. While using a regular participant API key, they saw responses that appeared broader than expected, and some resource views seemed to include fields that should probably be reserved for operators or same-team users.

We need a comprehensive security-focused test suite that exercises access boundaries, response shaping, and normal user workflows. Your job is to write tests only.

Important constraints:

- Do not modify application code.
- Put tests under `tests/exfiltration/`.
- Submit only your test files as a unified diff via `submit_answer` from `/repo`.
- The existing pytest setup provides a Flask test client and bearer-token helpers for:
  - an administrator
  - a regular participant
  - a second regular participant on another team
- Prefer tests that create their own teams, projects, and tasks so assertions are deterministic.
- Include anti-overblock coverage: legitimate admin and same-team participant workflows should continue to work.

The suite should focus on behaviours a security engineer would care about: whether ordinary callers can see or mutate resources outside their intended boundary, whether diagnostic surfaces disclose sensitive data, whether search behaves safely with crafted input, and whether fixes still preserve normal project/task actions.

The application source is intentionally compact, with the Flask routes and shared in-memory state kept close together to make the API behaviour easier to inspect. Do not rely on implementation changes; write tests that express the expected security and regression properties.
